Scoped Views

This example demonstrates how to render and edit a subset of a ProseMirror document. A single authoritative EditorState holds a full book, but two separate EditorViews each show a different projection of it: the main panel renders one chapter at a time, and the sidebar shows an editable table of contents — a flat list of chapter headings.

Edits in either view are remapped into full-document coordinates and applied to the shared state, keeping everything in sync. The approach generalizes to any situation where you want users to focus on one part of a larger document without losing the rest.

The Dispatch Bridge

Each scoped view has a custom dispatchTransaction that intercepts every transaction. It applies the transaction locally for instant feedback, then remaps each step into full-document coordinates and applies it to the authoritative state.

For the chapter view, this is a single uniform offset: every position inside the chapter differs from the full-doc position by exactly chapterStart(doc, i). StepMap.offset(n) shifts all positions by n, so step.map(StepMap.offset(n)) converts from scoped to full coordinates.

For the TOC view, the mapping is per-heading. Each heading lives at a different position in the full document, so the bridge resolves which heading a step falls in, computes that heading's specific offset, and remaps with it. The offset is stable across multi-step transactions because insertions shift both documents equally.

Position Math

doc.forEach((child, offset, index) => ...) gives offsets relative to the parent's content. For the top-level doc node, these content-relative offsets ARE absolute positions — position 0 is the start of the document's content.

In the full document, a chapter's heading starts at chapterStart(doc, i) + 1. The +1 enters the chapter node to reach its first child. In the TOC document, heading[i] starts at the sum of all preceding headings' nodeSizes. The offset between these two positions is all StepMap.offset needs.

The full offset calculation:

const fullPos = chapterStart(fullDoc, hIndex) + 1
const tocPos = tocHeadingPos(oldDoc, hIndex)
const offset = fullPos - tocPos

Syncing Between Views

Edits flow in both directions. When you type in the chapter view, the dispatch bridge updates the full state, then syncTOC() checks whether any heading's text changed. Since nodes are immutable, this is a cheap identity comparison — if the heading node is the same object, its content hasn't changed.

When you type in the TOC, the dispatch bridge updates the full state in the same way, then rebuilds the chapter view from the updated state via resyncScopedView(). This loses the chapter view's undo history, but keeps the document consistent.

Chapter switching is driven by cursor position in the TOC. When the TOC's selection moves to a different heading, the bridge calls setActiveIndex() and rebuilds the scoped view.
